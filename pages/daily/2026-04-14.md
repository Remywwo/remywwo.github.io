## 核心观点

过去一年，我们与数十个行业团队合作构建 LLM 智能体。**最成功的实现往往不是依赖复杂的框架或专用库，而是采用简单、可组合的架构模式。** 这篇文章总结了 Anthropic 在构建有效智能体过程中的实战经验。

## 几个重要区分

**Workflows（工作流）**：通过预定义的代码路径来编排 LLM 和工具，属于确定性执行。

**Agents（智能体）**：由 LLM 动态指导自己的流程和工具使用，任务执行路径不固定。

两者不是对立关系，而是适用于不同场景的工具。

## 何时使用、何时不用智能体

LLM 应用开发的一个基本原则是：**能用简单方式解决就别上智能体。** 智能体方案通常在延迟和成本上有所牺牲，换取更好的任务表现——需要评估这个 trade-off 是否值得。

对于复杂任务，workflows 更适合流程固定、需要高可预测性的场景；agents 则适合需要模型动态决策的开放性问题。大多数场景下，优化单次 LLM 调用 + 检索 + in-context examples 就足够了。

## 框架的使用建议

常见框架如 Claude Agent SDK、AWS Strands Agents、Rivet、Vellum 等，简化了调用 LLM、定义工具、链式调用等底层操作，但同时引入了额外的抽象层，可能掩盖底层的 prompt 和响应，增加调试难度。

建议：**先用 LLM API 直接实现**，很多模式只需几行代码。如果要用框架，必须理解底层代码——对"黑箱"配置的常见假设是导致错误的重要来源。

## 六种核心模式

### 1. Prompt Chaining（链式提示）

将任务拆分为**固定顺序的子步骤**，每个 LLM 调用处理前一个的输出。中间步骤可加编程检查（gate）。

适用场景：任务能清晰拆解为固定子任务，用延迟换准确率。

例子：写文案 → 翻译成其他语言；写大纲 → 检查大纲 → 写正文。

### 2. Routing（路由）

**分类输入并引导到专门的后续处理流程**，实现关注点分离和更专业的 prompt 优化。

适用场景：任务有明确分类，且分类可以准确执行（LLM 或传统算法均可）。

例子：客服工单分流到不同处理流程；简单问题用小模型（如 Haiku），复杂问题用大模型（如 Sonnet）。

### 3. Parallelization（并行化）

LLM **同时处理子任务，输出结果通过程序聚合**，两种形式：
- **Sectioning**：将任务拆分为独立子任务并行执行
- **Voting**：同一任务执行多次获取多样化输出

适用场景：子任务可并行加速，或需要多角度/高置信度结果。

例子：Sectioning——一个模型处理用户查询，另一个筛查不当内容；Voting——多个 prompt 审查代码漏洞。

### 4. Orchestrator-Workers（编排器-工作者）

**中心 LLM 动态分解任务**，委托给子 LLM，再汇总结果。关键区别在于灵活性——子任务不是预定义的，而是由编排器根据输入动态决定。

适用场景：复杂任务，无法预先预测所需的子任务（如代码修改，可能涉及多个文件的不同改动）。

### 5. Evaluator-Optimizer（评估器-优化器）

**一个 LLM 生成响应，另一个提供评估和反馈，循环迭代**。

适用场景：有明确的评估标准，迭代优化能带来可衡量的价值。

例子：文学翻译（评估器指出译者遗漏的细微差别）；复杂搜索需要多轮搜索和分析。

### 6. Agents

当 LLM 在关键能力上成熟时——理解复杂输入、推理和规划、可靠使用工具、从错误中恢复——agents 开始进入生产环境。

**工作流程**：接收人类命令或交互式讨论 → 明确任务后独立规划和操作 → 在关键节点暂停等待人类反馈 → 任务完成或达到停止条件时结束。

Agents 的实现通常不复杂，本质上就是 **LLM + 工具 + 环境反馈循环**。因此工具的设计和文档非常关键。

适用场景：**开放性问题，无法预测所需步数，无法硬编码固定路径**。由于自主性带来更高成本和误差累积风险，建议在沙盒环境中充分测试，并配备相应的护栏。

## 三大核心原则

Anthropic 建议实施智能体时遵循：

1. **保持设计简洁**
2. **优先透明度**：显式展示智能体的规划步骤
3. **精心设计 Agent-Computer Interface（ACI）**：通过完善的工具文档和测试，让工具对模型来说易于理解和使用

## 工具设计的实践建议

工具格式的选择影响 LLM 编写难度（生成 diff 比重写整个文件更复杂；JSON 内嵌代码比 markdown 代码块需要更多转义）。

建议原则：
- 给模型足够 token 空间"思考"，避免陷入死角
- 格式尽量接近互联网上自然出现的文本形式
- 避免格式化"开销"（如精确计算代码行数、字符串转义等）

**投入设计 Agent-Computer Interface 的精力，应相当于设计 Human-Computer Interface 的精力**。具体做法：
- 站在模型角度审视工具定义是否清晰
- 用描述性参数名和文档，类比为团队新人写文档
- 在 workbench 中大量测试，观察模型常犯的错误并迭代
- 通过参数设计减少误用（poka-yoke 思维）

**一个关键案例**：SWE-bench 智能体开发中，团队发现模型在离开根目录后使用相对路径会出错。解决方案：将工具改为**始终要求绝对路径**，此后模型使用再无问题。
